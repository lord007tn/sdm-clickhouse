# SDM ClickHouse v0.2 Feature Execution Plan

Status: Planning only (no new feature implementation in this plan)
Owner: Core desktop app team
Base branch: main
Date: 2026-02-27

## Scope Clarification

This plan reflects your latest direction:

- No Monaco dependency.
- Keep editor lightweight, but add SQL highlighting + autocomplete for ClickHouse objects.
- Implement all requested features after plan approval.

## Delivery Strategy

We will execute in ordered phases to reduce regression risk:

1. Foundation + architecture split
2. SQL productivity (lightweight editor + multi-statement runner)
3. Data safety + explorer UX
4. Observability + jobs + export
5. Schema/connection organization
6. SSH tunnel lifecycle

Each feature includes: objective, implementation details, data/API impact, acceptance criteria, tests, and risks.

## Feature 1: Lightweight SQL Script Editor (Highlight + Autocomplete)

Size: L

### Objective

Replace the plain textarea behavior with a lightweight SQL editing experience that supports:

- Syntax highlighting
- Table/column/function autocomplete
- Fast typing and low startup overhead

### Proposed Approach

- Use CodeMirror 6 (modular, lighter than Monaco).
- Add SQL language support with custom completion source.
- Build completion providers from current schema cache:
  - databases
  - tables
  - columns for selected table/database
  - ClickHouse keyword/function list
- Keep existing tab model; editor is per-tab content only.

### Backend/API Impact

- No new required backend commands for first iteration.
- Optional later: add `schema_list_functions` command for server-driven function suggestions.

### UI/UX Changes

- Replace `Textarea` in query panel with editor component.
- Shortcut support:
  - `Ctrl/Cmd+Enter`: run current statement or selected SQL
  - `Shift+Enter`: newline
- Keep current query tab persistence (localStorage).

### Acceptance Criteria

- Editor loads quickly in desktop runtime.
- Syntax highlighting works for SQL keywords/strings/comments.
- Autocomplete suggestions appear for schema items and keywords.
- Existing query execution flow still works unchanged.

### Test Plan

- Unit: completion source mapper from schema state.
- E2E: type SQL, trigger completion, execute selected query.
- Regression: tab restore and query execution still pass.

### Risks

- Memory/perf if completion list grows on huge schemas.
- Mitigation: debounce and scope suggestions by active database.

## Feature 2: Multi-Statement Runner with Per-Statement Results/Logs

Size: L

### Objective

Allow script execution with multiple statements and clear per-statement outcomes.

### Proposed Approach

- Add SQL splitter in backend with awareness of strings/comments.
- Execute statement-by-statement with generated child query IDs.
- Stop-on-error mode (default), with optional continue-on-error mode.
- Return structured result list:
  - statement index/text
  - success/failure
  - duration
  - row count/stats
  - error message

### Backend/API Impact

- New command: `query_execute_script(request)`
- New types:
  - `ScriptRequest`
  - `ScriptStatementResult`
  - `ScriptExecutionResult`

### UI/UX Changes

- Add script mode toggle in query toolbar.
- Add execution panel showing each statement status.
- Clicking a failed statement focuses that SQL block.

### Acceptance Criteria

- Scripts with 2+ statements execute in order.
- Failure clearly identifies failing statement.
- User can rerun only failed statement.

### Test Plan

- Backend tests for splitter + execution ordering.
- E2E: mixed success/failure script scenarios.

### Risks

- SQL splitting edge cases.
- Mitigation: conservative parser rules + documented limitations.

## Feature 3: Better Data Explorer (Column Filter/Sort/Limit without SQL)

Size: M

### Objective

Enable interactive browsing of table data without manually writing SQL.

### Proposed Approach

- Add explorer panel bound to selected table.
- UI controls:
  - column filters (equals/contains/range)
  - sort by one or more columns
  - limit/page size
- Generate safe SQL in frontend helper (or backend helper command).

### Backend/API Impact

Option A (faster): reuse existing `query_execute` with generated SQL.
Option B (safer): add `table_explore(request)` backend builder.

### UI/UX Changes

- New “Explore” section in table pane.
- Show generated SQL preview (read-only).

### Acceptance Criteria

- User can filter/sort/limit and fetch results from selected table.
- Generated SQL is visible and deterministic.

### Test Plan

- Unit: filter->SQL generator.
- E2E: apply filters and verify results change.

### Risks

- Query complexity growth.
- Mitigation: constrain operators in v1.

## Feature 4: Safe Mutation Flow (Preview SQL + Key-based Targeting)

Size: M

### Objective

Make updates/deletes safer than current row-wide predicates.

### Proposed Approach

- Introduce explicit mutation wizard:
  1. select target table
  2. choose key columns (primary/order key inferred when possible)
  3. preview affected rows count
  4. show generated SQL
  5. confirm with token
- Replace ad-hoc broad matching where possible with key-based predicates.

### Backend/API Impact

- Add key metadata helper command (if needed): `schema_table_keys`.
- Add mutation preview response containing canonical SQL fingerprint.
- Validate fingerprint/token on execute.

### UI/UX Changes

- Replace generic dangerous prompts with typed wizard states.

### Acceptance Criteria

- Update/delete requires preview + explicit confirmation.
- Executed SQL matches previously previewed canonical statement.

### Test Plan

- Backend: preview/execute consistency tests.
- E2E: update/delete happy path and mismatch rejection.

### Risks

- Tables without stable key columns.
- Mitigation: fallback flow with stronger warning and narrow WHERE requirement.

## Feature 6: Query Jobs Center

Size: M

### Objective

Provide one place to monitor and control running/completed/failed queries.

### Proposed Approach

- Introduce jobs store keyed by query ID.
- Track lifecycle events from run/cancel actions.
- Support global cancel from jobs center.
- Desktop notification for completion/failure (optional toggle).

### Backend/API Impact

- Reuse `query_cancel`.
- Optional future: server polling command for long jobs.

### UI/UX Changes

- Add docked side panel or bottom drawer “Jobs”.

### Acceptance Criteria

- User sees active and recent jobs across tabs.
- Cancel action works from jobs center.

### Test Plan

- E2E: run long query, observe job state transitions.

## Feature 7: Result Export (CSV/JSON/Parquet + Copy with Headers)

Size: M

### Objective

Make result sharing/export first-class.

### Proposed Approach

- Add export menu in result toolbar.
- Formats:
  - CSV
  - JSON
  - Parquet (phase 2 inside this feature if dependency cost is high)
- Include options:
  - current page vs all fetched rows
  - include headers
  - delimiter for CSV

### Backend/API Impact

- Fast path: frontend export for current in-memory rows.
- Scalable path: backend stream export command for large result sets.

### UI/UX Changes

- Export button + file save dialog integration.

### Acceptance Criteria

- User can export visible result set to CSV/JSON reliably.
- Copy with headers works to clipboard.

### Test Plan

- E2E: export file exists + basic content sanity checks.

### Risks

- Large dataset memory pressure.
- Mitigation: stream mode for larger exports.

## Feature 10: Real SSH Tunnel Management (Start/Stop/Status)

Size: L

### Objective

Upgrade SSH from metadata-only to active tunnel lifecycle management.

### Proposed Approach

- Manage tunnel process per connection profile:
  - start
  - stop
  - health/status
  - restart on failure (configurable)
- Support auth modes:
  - key file
  - password (stored securely)
- Track local forwarded port and connection binding.

### Backend/API Impact

- New tunnel manager module in Rust.
- New commands:
  - `ssh_tunnel_start`
  - `ssh_tunnel_stop`
  - `ssh_tunnel_status`
- Persist minimal tunnel settings in metadata DB.

### UI/UX Changes

- Tunnel controls in connection card/dialog.
- Status badge: disconnected/connecting/connected/error.

### Acceptance Criteria

- User can start/stop tunnel and connect through it reliably.
- Tunnel status reflects real process state.

### Test Plan

- Integration tests with test SSH server (CI optional via docker).

### Risks

- Cross-platform SSH binary differences.
- Mitigation: detect environment and show actionable guidance.
